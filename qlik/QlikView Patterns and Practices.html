Experience Design

Having a consistent and standard layout facilitates user adoption and efficiency.

Sheets

Sheets should be designed to look best on the most common display through which the report will be viewed. Also, sheets should be designed to look at least passable on the smallest display through which the report will be viewed. Additionally, be sure to test the web view if a browser is a means by which the report will be viewed. If a sheet cannot be designed to look at least passable on all displays through which it will be viewed, making a separate sheet or report is advised.
Qlik Design Blog: One App to rule them all

Filters

Up to 2 sides of each sheet should have filters, top or bottom AND/OR left or right. Across each sheet, the side(s) used should be consistent, and the same filters should be located in the same place.
Since the header is generally larger than the footer, the top side is more suited for filters than the bottom side. Date/Time and other more universal (across multiple or all sheets) filters are well-suited to the top side, since the header does not have as much room to spare as the left or right side, and those filters usually take up little space and are few.
Since the size of some charts/tables is dynamic based on the amount of data, the left side is more suited for filters than the right side. Other and sheet-specific filters are best-suited for the left side, since there is plenty of room for them, and those filters are often many.
Qlik Design Blog: Left or Right side navigation?

Sheet Flow

Following a logical sequence of sheets encourages data discovery. If more than 1 Dashboard or 1 Reporting sheet seems beneficial, consider making multiple reports.
Qlik Design Blog: DAR Methodology

1 Dashboard Sheet

This sheet should be first. It should contain the shallowest level of detail, focusing on KPIs (Key Performance Indicators) with few filters (usually just Date/Time filters). The size of text and charts/tables should be fairly large, but the amount of text and the volume and number of charts/tables should be minimal.

1+ Analysis Sheet(s)

These sheets should contain a medium level of detail, each focused on drilling down into the data with various charts. They are usually themed around areas of the Dashboard sheet, user roles, or user tasks.

1 Reporting Sheet

This sheet should be last. It should contain the deepest level of detail, generally featuring a table showing the full volume of data that enables the user to make their own report by selecting which dimensions and facts they want to see.

0/1 Help Sheet(s)

This sheet should be before the Dashboard sheet or after the Reporting Sheet. It should contain meta detail (e.g.: creator, contact, version control info, last refresh date, user name, user role/permissions, etc...) for easy reference but is no substitute for proper documentation. If it would only contain technical details, it can instead be hidden (potentially via user roles/permissions) or it can be excluded and have its content placed in the Load Script to avoid confusing the end users.

Version Control

Being able to track, view, and merge changes to files and is incredibly useful, especially when more than 1 person works on the same project. This is just as true with QlikView, but the available options and their respective pros & cons are not immediately obvious.
Note that QlikView does not provide any built-in Version Control; it merely exposes itself in a versionable manner, hence the need for actual Version Control Software. Also, pre-existing QlikView reports can be switched to using Version Control with ease.
Additionally, QlikView tracks many details (e.g.: last active Load Script tab, etc...) that do not need versioning, so inspect all changes to potentially undo some. This will reduce the "noise" when viewing change history.

General 3rd Party Product (e.g.: PRJ Folder)

Required Version Control Software: Any (e.g.: Git, Mercurial, Subversion, Team Foundation Server, etc...)
Pros: Free, Flexible, Unified
Cons: Manual, Not Supported
QlikView Menu(s): None or File ⇒ Export & File ⇒ Import
When a |NAME|.qvw file and a |NAME|-prj folder are with the same folder, ...

  • Saving the QVW file automatically exports to the PRJ folder.
    • Data is not included in the export (and thus, import), saving space and isolating only what should be versioned.
  • Opening the QVW file (even if it's a blank or out-of-date QVW) automatically imports from the PRJ folder if the PRJ folder is not empty.
Using the Export Document Layout... & Import Document Layout menu commands accomplishes the same but allows for flexibility in PRJ folder name and location.

1st Party Product (e.g.: QlikView Source Control Interface)

The menu contains many commands:

  • Settings... is for choosing which Version Control Software used and specifying the mapping to it.
  • Add Project to Source Control... exports the project to the chosen Version Control Software at the specific mapping. This only needs to be done once per project.
  • Get Project from Source Control... imports the project from the chosen Version Control Software at the specific mapping. This only needs to be done once per project per location (working copy / branch).
  • Get Latest Version syncs the project with the last committed version. Resolving conflicts may be compulsory if you have pending changes.
  • Check In Pending Changes... commits all pending changes. Resolving conflicts may be compulsory if other changes were committed since you last issued the Get Latest Version menu command.
  • Undo Pending Changes... reverts all pending changes.

Deployment

Though QlikView neither needs nor has a build/publish process, deployment to different environments is usually a need. Along with this, packaging or collecting all the files upon which a report depends is very helpful but not strictly necessary. Note that QlikView does not expose itself 100% to version control, partially intentionally (in the case of variables) and partially unintentionally (I've seen mixed info on this, some of which may be outdated, so just test for yourself and investigate if something seems to have been excluded).

By Hand

Pros: Free, Flexible
Cons: Very Manual, Not Unified, Not Supported
This is the full process that the other methods simplify through consolidation and workarounds. Though far from ideal, it is not a terrible solution if deployments are rare or incremental.

  1. Delete the old folders at the destination, if they exist.
  2. Copy the new folders to the destination, adjusting for the environment as necessary.
  3. Create the QVW files, if they do not yet exist.
    • Due to the way security for Qlik Server currently works, the QVW files should never be deleted with regard to deployment unless you want to have to reenter the permissions.
  4. Open the QVW files.
  5. Save the QVW files.
  6. Close the QVW files.
  7. Delete the PRJ folders at the destination to improve performance when opening the QVW files in the future.
  8. Reload the QVW files via the Report Architecture Staging Step Scheduling.

General 3rd Party Product (e.g.: IDE Solution/Project)

Pros: Free, Flexible, Unified
Cons: Manual, Not Supported
By using one of these (e.g.: Visual Studio SLN, Eclipse PROJECT, etc...), an artificial build/publish process is introduced for compiling and automating deployment-specific steps. Since these do not support QlikView specifically, custom configuration and scripts need to be made and adjusted to make best use of this option.

Report Architecture

How a report is structured can greatly impact its maintainability.

External Resource Tracking

Certain external resources either are tracked poorly (with regard to the level of tracking a developer or owner would want) in QlikView, and there is often little-to-no ability to document outside of the Load Script. Defining these external resources in a separate file allows them to be more easily tracked and documented. Since QlikView can load from spreadsheets easily, a spreadsheet is a good option.

Variable

First of all, a variable should be used any time a given calculation appears twice. If the formula for a variable refers to other variables, it is more clear and less error-prone to place it after the variables it references.
QlikView does not yet version variables, so this is necessary for version control.

File

While this can certainly be done with the data files, it is primarily for files like images. Images, for instance, are often embedded directly in sheet objects, so having them all defined in 1 place along their intended purpose (in case the file moves and breaks the image) is neater. This also allows for easy association of files, like images, to records, in case they aren't or can't be stored in the data source.
Qlik Design Blog: Loading Images into QlikView

Connection String

If having many data sources requiring the CONNECT statement, storing those connection strings in a separate file can be good for organization and maintenance (and avoids cluttering the Shared Scripts if they are used in multiple places).

Staging Steps

For more complex reports with varied or incongruous data sources, stage the data from outside of the report itself. A goal is to have as little logic as possible in the report itself. This achieves separation of concerns in what is known as an ETL (Extract, Transform, Load) process.
A report can load 1 QVW file and any number of QVD files. Thus, creating QVD files is the primary method of implementing these steps. However, the final step is often best left to a QVW file.

ETL Steps

Only use the steps needed, generally 1 step for each concern in the ETL process.

  1. Extract: This is where the data is accessed. Output QVD files. This step can often be reused by other reports.
  2. Transform: This is where the data is related and formatted. Output QVD files. Take care in pre-aggregating data, lest discovery be inhibited.
  3. Load: This is where the data is combined. Output a QVW file.

Scheduling

Create a job chain on Qlik Server to run the staging steps in dependency order. This job chain often begins with a dummy trigger.

Shared Scripts

Another way to eliminate duplication and thus lessen maintenance is to package common Load Script code into separate QVS files for reuse within the Load Script of this or other reports. Always use $(MUST_INCLUDE=[Path\To\Script.qvs]), as $(INCLUDE=[Path\To\Script.qvs]) silently fails if the script does not exist.
Qlik Design Blog: The $(Include) which you $(Must_Include) into your toolkit

Data Modeling

How the data model is structured can have a significant impact upon QlikView's performance.

Galaxy/Constellation Schema

Dimension Tables: 1+
Fact Tables: 1+
This choice is either a Snowflake or Star Schema (the definitions I have found are either not consistent enough to prove reliable or make no mention of which) involving multiple fact grains/levels (in a hierarchical sense) or types in separate tables. Its suitability is most apparent when facts share multiple (but not all) dimensions. Use as needed, especially instead of mixing fact granularity or fusing unrelated fact types in a fact table.

Snowflake Schema

Dimension Tables: 1+, Normalized
Fact Tables: 1
This choice logically expresses dimensional hierarchies around a single fact grain or type with minimum redundancy.

Star Schema

Dimension Tables: 1+, Denormalized
Fact Tables: 1
This choice is the best for optimal performance with a single fact grain or type, though only necessary for the largest of data sets.

No Schema

Dimension Tables: 0
Fact Tables: 1+
While using 1 table (or 1 table for each fact grain or type) to store all of the data is possible and provides some benefits, the drawbacks far outweigh them. Do not use this, even if your data source uses this schema.

Data Connecting

The way QlikView handles connections between sets of data is quite unique, so explanation is necessary. This document does not explain Joins or Unions conceptually in depth, as many other resources do this adequately (such as SQLity: A Join A Day, noting that some of the details are specific to Microsoft SQL Server).

How many column names match? QlikView performs Condition
None CROSS JOIN any type of Join is specified
Some Natural OUTER JOIN none/automatic
All CONCATENATE none/automatic

Join

Can be thought of as: Adding the columns of a table to another table, potentially adding/removing rows as well.

QlikView Join Type SQL Equivalent
OUTER JOIN FULL OUTER JOIN & COALESCE([joined table 1].[key], [joined table 2].[key]) AS [key] for each key involved in the join
RIGHT JOIN RIGHT OUTER JOIN & Left keys not selected
LEFT JOIN LEFT OUTER JOIN & Right keys not selected
INNER JOIN INNER JOIN & USING
To prevent unintentional implicit Natural Joins, specify the the desired type of Join and both QUALIFY * and UNQUALIFY [the list of keys on which to join] when joining is desired, and specify just QUALIFY * when joining is not desired.
Semi Joins and Anti (Semi) Joins can be simulated with EXISTS([key]) and NOT EXISTS([key]), respectively, AND-ed for each key involved in the type of Outer Join (LEFT, RIGHT, or FULL) and followed by dropping the extra table if not a FULL OUTER JOIN. Using KEEP for Semi Joins may be a simpler alternative.
Though SQL and other relational tools default JOIN to INNER JOIN, QlikView defaults JOIN to FULL OUTER JOIN since it is associative. To avoid confusion, do not use just JOIN.
QlikView only supports simple Equi-Joins, meaning that it can only join based on equality (i.e.: no >, <, >=, <=, or !=/<>), and every key must match (i.e.: only basic AND logic).
Qlik Design Blog: Do you Qualify?

Map

Can be thought of as: Adding a column of a table to another table.
SQL Equivalent: DECODE, CASE, or scalar subquery
Its simplest use case is as a computationally faster alternative to joining to a reference table (or issuing an equivalent scalar subquery) for translating an id/key into a name/title/description, using DECODE or CASE to manually translate values in SQL, or cleansing data. Mapping can also be chained to produce complex translations.
Mapping assumes there is a 1:1 or M:1 relationship. Thus, if there is actually a 1:M relationship, it will return the first match it encounters, thinking it is the only one. This is generally not desired behavior, as an aggregation (e.g.: min, max, sum, etc...) is usually desired in a 1:M relationship. To account for this, GROUP BY the Key, and aggregate the Value in your MAPPING LOAD. To avoid glitches, use ONLY as the aggregation for all 1:1 relationships not enforced by a DB.
Some (usually those with little SQL experience) advocate for mapping to the exclusion of left and right joins due to the benefits (e.g.: speed, no potential for row duplication in a 1:M relationship, etc...), but they ignore the drawbacks (e.g.: silent failure on map name typo, potential for inconsistent results in a 1:M relationship, need for multiple mapping tables or a string-concatenated mapping table if mapping than 1 column for a given key, etc...). Both exist in QlikView for a reason, and the tool appropriate to the circumstance should be used.

  • APPLYMAP is the primary mapping function. When it does not find a match, it returns the 2nd parameter unless you specify a 3rd parameter. While returning the 2nd parameter can be useful while creating a report, it is rarely if ever desired in a finished report, so be sure to include the 3rd parameter.
  • MAP ... USING is a shortcut for reusing the same map. Be sure to UNMAP when done.
  • MAPSUBSTRING is like a complex "Find & Replace", mapping every occurrence of the value it finds within the supplied key. It can be useful for mapping a composite key if unable to map when it was still individual key fields.
  • Qlik Design Blog: Mapping ... and not the geographical kind
  • Qlik Design Blog: Don't join - use Applymap instead

Concatenate

Can be thought of as: Adding the rows of a table to another table.
SQL Equivalent: UNION ALL
To prevent unintentional implicit concatenation, specify CONCATENATE([table]) when concatenation is desired, and specify NOCONCATENATE when concatenation is not desired.
As for other set manipulation statements, QlikView does not have built-in UNION DISTINCT, INTERSECT, or MINUS/EXCEPT support. UNION DISTINCT can be accomplished using NOT EXISTS([column]) AND-ed for every column, but INTERSECT and MINUS/EXCEPT are not easily possible.

Generic

Can be thought of as: Denormalization. Flipping dimensions into columns across the diagonal.
SQL Equivalent: PIVOT
Especially is gotten from a NoSQL database (e.g.: entity-key-attribute-value), data is often in too normalized (usually 6th Normal Form). GENERIC can fix this, making such data more useful for analysis by bringing it to at most 5th Normal Form.
Qlik Design Blog: The Generic Load

Crosstable

Can be thought of as: Normalization. Flipping columns into dimensions across the diagonal.
SQL Equivalent: UNPIVOT
Especially if gotten from a spreadsheet, data is often too denormalized (anything below 3rd Normal Form). CROSSTABLE can fix this, making such data more useful for analysis by bringing it to at least 3rd Normal Form. Note that the data can be displayed in a pivot table as if it was still denormalized.
Due to being single-threaded, this is known for being slow on anything but simple datasets. So, do this in SQL, if possible.
Qlik Design Blog: The Crosstable Load

Selection Cardinality

A filter can have 0, 1, or many selections (0 is the same as if all values were selected). It is possible to enforce a "1 and only 1" selection requirement, which is useful in certain cases where no or multiple selections would not make sense for the data or presentation.

Logic Mode

Though QlikView's filtering is powerful and designed for the most common use case, it is nonetheless limited.
Within a single column, multiple selections operate with OR logic (Year = 2015 OR 2014). It is possible to switch to AND logic for individual columns, but this requires an extra linking table to be defined.
Across multiple columns, multiple selections operate with AND logic (Year = 2015 AND Month = Jan). It is possible to switch to OR logic for collections of columns, but this is a difficult and intensive process using advanced Set Analysis functionality. Additionally, OR logic for collections of columns is generally needed far less often than AND logic for individual columns.
These alternate logic modes change (or even reduce) the information expressed by the QlikView color scheme.

Focusing (a.k.a.: Contextual Brushing / Coordinated Highlighting)

In QlikView, clicking on a data point in a chart or table selects (or, filters on) the dimension values responsible for that fact. In other tools like Tableau, this action instead highlights (or, focuses on) those dimension values. QlikView can mimic this and other focusing behaviors, though circumventing QlikView's default behavior may be confusing to end users.

Comments

QlikView's script syntax uses C-style comments (//... & /*...*/), which means that QlikView script block comments (/*...*/) overlap with SQL syntax. Since QlikView LOADs look similar to SQL SELECTs, it is important to understand (for the purposes of SQL hints) that all text after the SQL keyword and until the semicolon is sent directly to the SQL engine to which the script is connected without any pre-processing. This also means that QlikView script inline comments (//...) in SQL will not cause compile-time errors but will cause run-time errors.

NULL

While in the Load Script QlikView generally handles different types of missing data like most databases, it is still important to learn this behavior. After the Load Script, NULLs behave differently in filters, metrics, chart/table dimensions, and chart/table measures.
Qlik Design Blog: NULL handling in QlikView

Dual

QlikView is unique in its handling of data types; specifically, it actually has none. Instead, every scalar value is a combination of a string (for display) and a number (for sorting). This is how QlikView automatically translates week day numbers and month numbers into text. Both "sides" of a value can be used in calculations, so there is no loss in taking advantage of this intentional design decision. In fact, there are a host of use cases for manipulating the dual format. Be aware, though, that the dual format does not always behave as a scalar value is expected to behave.

Master Calendar

In nearly any reporting effort, master calendars are incredibly useful for maintaining consistency when implementing and tracking special time period as defined by business rules (e.g.: fiscal year, pay period, work week, workday/holiday, etc...). Due to QlikView's associative nature, date and time need to be handled with care and consideration. While the "one" (canonical for all date/time fields), the "many" (1 for each date/time field), or both master calendar strategies can and should be used, time, if needed, should always be separated into its own fields and tables to reduce memory overhead.
